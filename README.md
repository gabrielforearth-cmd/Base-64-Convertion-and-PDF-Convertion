# PDF Processing and Base64 Text Recovery Utilities

This project contains two Python utilities for document processing:

1. A DOCX-to-PDF workflow that converts accounting documents, attempts PDF/A-3 conversion, adds an image page, and generates processing logs.
2. A Base64 text recovery utility that tests multiple character encodings and generates a PDF containing the original and decoded values.

## Project Structure

```text
pdf-processing-utilities/
├── generate_accounting_pdf.py
├── decode_base64_to_pdf.py
├── Contabilidade.png
├── DejaVuSans.ttf
├── inputs/
├── outputs/
└── README.md
```

The two code blocks should not be placed in the same Python file.

Each script has separate imports, responsibilities, and execution entry points.

# 1. Accounting PDF Generator

## Purpose

`generate_accounting_pdf.py` processes every DOCX file from an input directory.

For each document, the script:

1. Converts the DOCX file to PDF using LibreOffice.
2. Attempts to save the PDF using PDF/A-3B compliance.
3. Creates a PDF page containing an image.
4. Appends the image page to the converted document.
5. Saves the final PDF in the output directory.
6. Simulates a workflow status update.
7. Writes a simulated chat message to the application log.

## DOCX Conversion

The script uses LibreOffice in headless mode:

```python
cmd = [
    "libreoffice",
    "--headless",
    "--convert-to",
    "pdf",
    docx_path,
    "--outdir",
    os.path.dirname(pdf_path)
]
```

The command does not open the LibreOffice graphical interface.

A document named:

```text

```

is converted by LibreOffice into:

```text
Accounting_Report.pdf
```

## Important Temporary Filename Detail

The script defines the expected temporary PDF path as:

```python
temp_pdf = os.path.join(
    output_folder,
    f"{base_name}_temp.pdf"
)
```

However, LibreOffice normally preserves the source base filename during conversion.

For example, converting:

```text
Accounting_Report.docx
```

usually creates:

```text
Accounting_Report.pdf
```

and not:

```text
Accounting_Report_temp.pdf
```

Therefore, the current implementation may report a path that LibreOffice did not actually create.

A safer conversion function should identify the file generated by LibreOffice and rename it when necessary.

Example:

```python
def docx_to_pdf_linux(docx_path, pdf_path):
    output_directory = os.path.dirname(pdf_path)
    source_base_name = os.path.splitext(
        os.path.basename(docx_path)
    )[0]

    generated_pdf = os.path.join(
        output_directory,
        f"{source_base_name}.pdf"
    )

    try:
        command = [
            "libreoffice",
            "--headless",
            "--convert-to",
            "pdf",
            docx_path,
            "--outdir",
            output_directory
        ]

        subprocess.run(
            command,
            check=True,
            capture_output=True,
            text=True
        )

        if not os.path.exists(generated_pdf):
            logging.error(
                "LibreOffice did not generate the expected PDF: %s",
                generated_pdf
            )
            return None

        if os.path.abspath(generated_pdf) != os.path.abspath(pdf_path):
            os.replace(generated_pdf, pdf_path)

        logging.info(
            "DOCX converted to PDF: %s",
            pdf_path
        )

        return pdf_path

    except subprocess.CalledProcessError as error:
        logging.error(
            "Error converting DOCX to PDF: %s",
            error.stderr or error
        )
        return None
```

## PDF/A-3 Conversion

The script attempts to save the converted PDF using:

```python
with pikepdf.open(pdf_path) as pdf:
    pdf.save(
        temp_pdf,
        compliance=pikepdf.PdfaCompliance.PDFA_3B
    )
```

The intended compliance level is:

```text
PDF/A-3B
```

PDF/A is an archival PDF standard designed for long-term document preservation.

### Important Limitation

Saving a PDF with a compliance argument alone may not be sufficient to create a fully valid PDF/A document.

PDF/A validation can require:

* Embedded fonts
* An output color profile
* Correct XMP metadata
* Supported transparency
* Valid color spaces
* Removal of prohibited features
* Correct PDF/A identification metadata

The generated file should be validated with an independent PDF/A validator before being used in a compliance-sensitive workflow.

## Adding the Image

The script creates an in-memory PDF page using ReportLab:

```python
packet = io.BytesIO()
canvas_object = canvas.Canvas(
    packet,
    pagesize=A4
)

canvas_object.drawImage(
    image_path,
    50,
    700,
    width=200,
    height=100
)

canvas_object.save()
```

The image is positioned using PDF coordinates.

```text
X position: 50
Y position: 700
Width: 200
Height: 100
```

The origin of a PDF page is normally the bottom-left corner.

## Image Behavior

The current implementation does not place the image over an existing document page.

Instead, it creates a new PDF page and appends that page to the end of the original PDF:

```python
writer.add_page(img_pdf.pages[0])
```

The resulting document contains:

1. All original converted pages
2. One additional page containing the image

To place the image on the first existing page, use page merging instead:

```python
overlay_page = img_pdf.pages[0]
first_page = original_pdf.pages[0]

first_page.merge_page(overlay_page)
writer.add_page(first_page)

for page in original_pdf.pages[1:]:
    writer.add_page(page)
```

## Important PDF/A Consideration

The image page is added after the PDF/A conversion step.

This means the resulting final PDF is modified after the attempted compliance conversion:

```text
PDF conversion
    ↓
PDF/A conversion
    ↓
New page added
```

The final PDF may therefore no longer comply with PDF/A-3.

For a more reliable workflow, perform PDF/A conversion after all pages and images have been added:

```text
DOCX conversion
    ↓
Add or merge image
    ↓
Generate complete PDF
    ↓
Convert complete document to PDF/A
    ↓
Validate PDF/A
```

# 2. Base64 Text Recovery and PDF Report

## Purpose

`decode_base64_to_pdf.py` receives two Base64 strings, attempts to decode their text using multiple character encodings, and creates a PDF report containing:

* The first original Base64 string
* The first decoded text
* The encoding selected for the first string
* The second original Base64 string
* The second decoded text
* The encoding selected for the second string

The utility is designed to help investigate encoding problems, especially with German characters.

## Supported Decoding Attempts

The script tests the following encodings:

```python
encodings = [
    "utf-8",
    "cp1252",
    "latin-1"
]
```

It also attempts recovery paths described as:

```text
cp1252 → UTF-8
latin-1 → UTF-8
```

When URL-encoded sequences are present, the script also attempts URL decoding.

Example:

```text
Gr%C3%BC%C3%9Fe
```

can become:

```text
Grüße
```

## German Character Detection

The script prioritizes candidates containing:

```text
ä ö ü Ä Ö Ü ß
```

These characters are defined by:

```python
GERMAN_CHARS = "äöüÄÖÜß"
```

## Requirements

Install ReportLab with:

```bash
pip install reportlab
```

The following modules come from the Python standard library:

* `base64`
* `re`
* `urllib.parse`
* `textwrap`

## Unicode Font

The script attempts to register:

```text

```

using:

```python

```





## Base64 Decoding

The script first converts the Base64 input to bytes:

```python
raw = base64.b64decode(
    b64_str,
    validate=True
)
```

Using `validate=True` means the input must contain valid Base64 characters and padding.

Invalid input can raise:

```text
binascii.Error
```

## Encoding Candidates

Each successfully decoded candidate is stored with a label:

```python
results.append(
    (
        encoding,
        raw_bytes.decode(encoding)
    )
)
```

Possible labels include:

```text
utf-8
cp1252
latin-1
cp1252->utf-8
latin-1->utf-8
```

When URL decoding is performed, the label receives:

```text
+url-decoded
```

Example:

```text
utf-8+url-decoded
```

## Candidate Scoring

The script selects the candidate with the lowest score.

The score penalizes characters associated with broken encoding:

```text
Ã
�
%
■
```

It also penalizes unexpected control characters.

German characters reduce the score, making the candidate more likely to be selected.

The scoring logic is:

```python
def score(candidate):
    text = candidate[1]

    bad = sum(
        character in "Ã�%�■"
        for character in text
    )

    bad += sum(
        ord(character) < 32
        and character not in "\n\r\t"
        for character in text
    )

    good = sum(
        character in GERMAN_CHARS
        for character in text
    )

    return bad - (2 * good)
```

The selected result is:

```python
best = min(
    fixed_candidates,
    key=score
)
```

## URL-Decoding Detection

The script checks for percent-encoded values with:

```python
re.search(
    r"%[0-9A-Fa-f]{2}",
    text
)
```

Examples include:

```text
%C3%A4
%C3%B6
%C3%BC
```

The utility then calls:

```python
unquote(text)
```

## Layout Limitation

The script uses fixed vertical positions:

```python
800
700
600
500
```

The `write_section` function does not return the final vertical position after writing wrapped lines.

When a string contains many lines, one section may overlap the following section.

A more reliable implementation should:

* Calculate the number of rendered lines
* Update the Y position dynamically
* Create a new page when space is insufficient

## Example of Dynamic Positioning

```python
def write_section(
    canvas_object,
    title,
    content,
    x,
    y,
    font
):
    canvas_object.setFont(
        "Helvetica-Bold",
        14
    )

    canvas_object.drawString(
        x,
        y,
        title
    )

    canvas_object.setFont(
        font,
        12
    )

    wrapped_lines = []

    for source_line in content.splitlines():
        wrapped_lines.extend(
            textwrap.wrap(
                source_line,
                width=80
            ) or [""]
        )

    current_y = y - 20

    for line in wrapped_lines:
        if current_y < 50:
            canvas_object.showPage()
            canvas_object.setFont(font, 12)
            current_y = 800

        canvas_object.drawString(
            x,
            current_y,
            line
        )

        current_y -= 15

    return current_y - 20
```

Usage:

```python
current_y = 800

current_y = write_section(
    canvas_object,
    "String 1 - Encoded:",
    base64_1,
    50,
    current_y,
    font_body
)

current_y = write_section(
    canvas_object,
    f"String 1 - Decoded [{how1}]:",
    decoded_1,
    50,
    current_y,
    font_body
)
```

## Error Handling Recommendation

The original code does not handle invalid Base64 input.

A safer version can use:

```python
import binascii


try:
    decoded_text, method = fix_text_from_base64(
        base64_value
    )

except binascii.Error:
    print("Invalid Base64 value.")
    return
```

Empty candidate lists should also be handled:

```python
if not fixed_candidates:
    raise ValueError(
        "The Base64 content could not be decoded "
        "using the supported encodings."
    )
```

## Encoding Recovery Limitations

The scoring algorithm is heuristic.

It does not guarantee that the selected text is correct.

Potential problems include:

* Non-German text may receive the wrong candidate
* Valid percent symbols are penalized
* A Latin-1 candidate always succeeds for arbitrary bytes
* CP1252 and Latin-1 may produce visually plausible but incorrect text
* The presence of German characters does not prove the encoding is correct
* Double-encoded text may require additional transformation steps

The selected result should be reviewed manually when accuracy is important.

# Important Technical Notes

## PyPDF2 and pypdf

The accounting script imports:

```python
from PyPDF2 import PdfReader, PdfWriter
```

`PyPDF2` and `pypdf` are related packages, but modern projects generally use the maintained `pypdf` package.

A modern import would be:

```python
from pypdf import PdfReader, PdfWriter
```

Install it with:

```bash
pip install pypdf
```

Avoid installing and mixing both libraries unless required by the project.

## Image added as a page

The accounting script appends the image as a separate page. It does not stamp the image onto every page or onto the first page.

## PDF/A after modification

Adding a page after converting to PDF/A can invalidate the archival compliance of the final file.

## LibreOffice output filename

LibreOffice determines the generated filename from the DOCX filename. The target path argument is used only to determine the output directory.

## Font licensing

The `DejaVuSans.ttf` file must be obtained and distributed according to its corresponding license.

## Security

Only process trusted documents when possible.

Malformed DOCX, PDF, image, or font files may cause processing failures in external libraries and applications.

## Limitations

### Accounting utility

* Requires LibreOffice to be installed and available in `PATH`
* Processes only DOCX files
* Processes only the top level of the input directory
* Does not recursively process subdirectories
* Appends the image as a new page
* Uses fixed image position and dimensions
* Does not validate the final PDF/A compliance
* Simulated status and chat functions do not update an external system
* Temporary files are not deleted automatically
* LibreOffice filename behavior must be handled carefully

### Base64 utility

* Accepts exactly two strings
* Uses interactive terminal input
* Supports only a limited set of encoding candidates
* Uses a heuristic to select the best result
* Fixed page positions may cause overlapping text
* The output filename is fixed
* Invalid Base64 input is not handled in the original version
* Helvetica fallback may not display all Unicode characters
* Long content may exceed the page boundaries
